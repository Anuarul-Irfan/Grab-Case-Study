# AI Reflection

I used ChatGPT Work/Codex as a design and implementation partner and GitHub Models as the proposed runtime category-classification service. ChatGPT helped me inspect the brief, data dictionary, SQLite reference data, submission data, and the three submission files. I also used it to challenge the category strategy, implement the Python pipeline, produce tests, and review operational risks. The runtime code uses the Azure AI Inference SDK with GitHub Models, with the endpoint, model, and batch size kept configurable.

I delegated mechanical work and adversarial review to AI, including enumerating distinct category descriptions, drafting validation code, constructing test cases, and checking whether the design satisfied each stated rule. I retained the main decisions: use an LLM rather than a fixed vocabulary for multilingual future inputs, classify unique values in batches, persist reviewed results, keep the reference database authoritative, and reject uncertain classifications rather than force a category. I chose the GitHub Models endpoint and GPT-5 integration.

One important limitation is that the AI had access to the submission data while helping construct and review the classification cache. The supplied sample run was then resolved using that cache and did not require a live token. This means the run demonstrates the pipeline’s processing, caching, validation, and output behaviour, but it is not an independent test of the model’s classification performance. Because the inputs were seen before execution, using the reviewed cache introduced evaluation leakage and would make any claim about model accuracy unreliable.

A fair evaluation would require separating development data from evaluation data. The existing cache should be cleared, and the live model should classify previously unseen descriptions without prior AI or human access to the expected results. Its predictions should be recorded before comparison with independently reviewed labels. Corrections could then be added to the persistent cache only after the evaluation is complete. This would preserve the practical benefits of caching without allowing advance knowledge of the test data to influence the results.

Another useful case where the AI was misleading concerned hardcoding. It initially said that an exact alias dictionary would be prohibited. After I reread the brief and challenged that claim, we established the more accurate distinction: the canonical reference lists must not be hardcoded, but the brief explicitly permits a rules or keyword approach. This changed how I documented the decision. The LLM design is a deliberate scalability choice, not something the exercise requires.

I did not accept generated code without verification. I checked the database tables with SQL, counted all 195 inputs and 43 distinct descriptions, ran the pipeline end to end, inspected the rejection distribution, audited every duplicate group, and ran automated tests. These checks validate the implementation and its handling of the supplied data, but they should not be interpreted as a blind assessment of model quality. Model output is treated as untrusted input: Python verifies its JSON structure, completeness, unique item IDs, status consistency, and category IDs against the current SQL results before a classification can be cached or written to output.

The largest long-term risk is not token cost but persistent error, because a bad model decision can be reused indefinitely. Production improvements would therefore include a human-review queue, correction history, quality metrics by language, and periodic evaluation against a separately maintained, reviewed multilingual dataset. Once enough independently reviewed examples accumulate, I would train a smaller local classifier and use the LLM only for new or ambiguous cases. This would make the cache an incremental source of labelled data while maintaining a clear separation between operational memory and evaluation evidence.
